# 텀프로젝트 제안서

## 1. 프로젝트 개요

### 1.1 주제명

스마트 주차장 빈자리 감지 및 차량 입차 제어 시스템

### 1.2 팀 구성원 정보

- 정보컴퓨터공학부 202055575 이동근

### 1.3 개발 배경 및 필요성

일반적인 소규모 주차장에서는 운전자가 직접 빈자리를 확인해야 하며, 주차 공간이 가득 찬 상황에서도 차량이 계속 진입하려고 시도할 수 있다. 이러한 상황은 불필요한 대기 시간, 주차장 내부 혼잡, 입구 정체를 유발할 수 있다. 본 프로젝트는 이러한 문제를 단순화된 미니 주차장 모델로 구현하여, 주차면의 점유 여부를 실시간으로 감지하고 빈자리가 있을 때만 차량 입차를 허용하는 시스템을 설계하는 것을 목표로 한다.

본 시스템은 NUCLEO-F439ZI 보드와 uC/OS-III를 기반으로 하며, 초음파 거리 센서 2개를 이용하여 2개의 주차칸 점유 상태를 판단한다. 입구에는 적외선 장애물 감지 센서를 배치하여 차량 입차 요청을 감지하고, 빈자리가 존재할 경우 서보모터를 이용하여 차단기를 개방한다. 만차 상태에서는 차단기를 열지 않고 2색 LED와 부저를 통해 입차 불가 상태를 알린다. 출차는 별도의 출구 센서를 두지 않고 관리자 버튼 입력으로 출차 이벤트만 처리하며, 실제 주차칸 점유 여부와 빈자리 수는 초음파 센서 측정 결과로만 갱신한다. 1602 I2C Character LCD에는 현재 빈자리 수, 각 주차칸 상태, 차단기 상태를 16자 x 2줄 형식으로 요약 표시한다.

본 프로젝트의 핵심은 단순히 센서 값을 읽고 출력 장치를 제어하는 것이 아니라, 여러 기능을 uC/OS-III의 독립적인 태스크로 분리하고 태스크 간 통신을 통해 동작시키는 것이다. 주차면 감지, 입구 차량 감지, 버튼 기반 출차 이벤트 처리, 상태 판단, 차단기 제어, LCD 표시, 경고 출력, USART 로그 출력을 각각 역할별로 나누어 실시간 임베디드 시스템의 구조를 구현한다. 기본 구현은 주차칸 2개로 진행하되, 센서 배열과 상태 배열의 크기를 조정하면 주차칸 4개까지 확장할 수 있도록 설계한다.

## 2. 프로젝트 목표

본 프로젝트의 최종 목표는 RTOS 기반으로 동작하는 스마트 주차장 제어 시스템을 구현하는 것이다. 구체적인 목표는 다음과 같다.

1. HC-SR04 초음파 거리 센서 2개를 이용하여 2개의 주차칸에 차량이 있는지 판단한다. 각 센서는 서로 간섭하지 않도록 순차적으로 측정하며, 측정된 거리값을 기반으로 주차칸 상태를 `EMPTY` 또는 `OCCUPIED`로 분류한다. 구현 구조는 센서 수를 상수로 관리하여 최대 4개의 주차칸까지 확장 가능하도록 설계한다.
2. 적외선 장애물 감지 센서를 이용하여 입구 차량 접근 여부를 감지한다. 차량 접근 이벤트가 발생하면 현재 빈자리 수를 확인하고, 빈자리가 있으면 차단기를 개방하고, 만차 상태이면 차단기를 닫은 상태로 유지한다.
3. 금속기어 서보모터를 이용하여 주차장 입구 차단기를 제어한다. 서보모터는 닫힘 각도와 열림 각도를 가지며, 입차 허용 조건이 만족되었을 때 일정 시간 동안 차단기를 개방한 후 다시 닫는다.
4. 1602 I2C Character LCD를 이용하여 사용자가 현재 주차장 상태를 쉽게 확인할 수 있도록 한다. LCD는 16자 x 2줄, 총 32칸으로 제한되므로 전체 주차 공간 수, 현재 빈자리 수, 각 주차칸 상태, 차단기 상태를 짧은 약어 형태로 표시한다.
5. 2색 LED와 수동 부저를 이용하여 입차 가능 상태와 만차 상태를 직관적으로 알린다. 입차 가능 상태에서는 초록색 LED를 표시하고, 만차 또는 입차 불가 상태에서는 빨간색 LED와 부저를 사용한다.
6. uC/OS-III의 메시지 큐, 세마포어, 임계 구역 등을 사용하여 태스크 간 데이터를 전달하고 공유 자원을 보호한다. 이를 통해 단일 `while` 루프 방식보다 명확한 구조와 안정적인 실시간 동작을 구현한다.
7. 버튼 입력을 이용하여 출차 또는 관리자 수동 처리 이벤트를 발생시킨다. 출차 버튼은 주차칸 상태를 직접 변경하지 않고, 출차 이벤트와 로그를 발생시키는 역할만 수행한다. 차량이 실제로 빠져나간 뒤 초음파 센서 측정값이 변화하면 `ParkingManagerTask`가 해당 주차칸을 `EMPTY`로 갱신한다.
8. USART 로그를 통해 센서 측정값, 주차칸 상태 변화, 입구 차량 감지, 버튼 기반 출차 이벤트, 차단기 개폐, 오류 상태 등을 확인할 수 있도록 한다.

## 3. 시스템 구성

### 3.1 하드웨어 구성

본 프로젝트에서 사용하는 주요 하드웨어는 다음과 같다.

| 하드웨어 | 수량 | 용도 | 설계상 주의점 |
|---|---:|---|---|
| NUCLEO-F439ZI | 1 | 전체 시스템 제어 및 uC/OS-III 실행 | 기존 수업 프로젝트의 BSP와 핀 충돌 확인 필요 |
| HC-SR04 초음파 거리 센서 | 2 | 각 주차칸 점유 여부 감지 | 센서 간 간섭 방지를 위해 순차 측정 필요. 구조상 4개까지 확장 가능 |
| 적외선 장애물 감지 센서 | 1 | 입구 차량 접근 감지 | 입력 신호의 채터링 또는 반복 감지 처리 필요 |
| 금속기어 서보모터 | 1 | 차단기 열림/닫힘 제어 | 별도 5V 전원 사용 및 GND 공통 연결 필요 |
| 1602 I2C Character LCD | 1 | 주차장 상태 요약 표시 | 16자 x 2줄 제한에 맞춰 짧은 문구 사용, I2C 주소 확인 필요 |
| 2색 LED 모듈 | 1 | 입차 가능/만차 상태 표시 | LED 상태 정의를 명확히 해야 함 |
| 수동 부저 모듈 | 1 | 만차 또는 동작 피드백 알림 | GPIO 또는 PWM 방식 결정 필요 |
| 버튼 모듈 | 1 | 출차 이벤트 및 관리자 수동 제어 | 디바운싱 필요. 점유 상태는 직접 변경하지 않음 |

NUCLEO-F439ZI 보드는 전체 시스템을 제어하는 메인 컨트롤러로 사용한다. uC/OS-III가 동작하며, 각 센서 입력과 출력 장치를 제어한다.

HC-SR04 초음파 거리 센서 2개는 각 주차칸에 하나씩 배치한다. 센서의 Trigger 핀은 STM32의 GPIO 출력에 연결하고, Echo 핀은 STM32의 GPIO 입력에 연결한다. HC-SR04의 Echo 출력이 5V일 수 있으므로, STM32 입력 핀 보호를 위해 1kΩ과 2kΩ 저항을 이용한 전압 분압 회로를 구성한다. 코드에서는 주차칸 수를 상수로 관리하여 센서와 핀 정의를 추가하면 4개 주차칸으로 확장할 수 있도록 한다.

적외선 장애물 감지 센서 1개는 주차장 입구에 배치한다. 차량 모형 또는 물체가 입구에 접근하면 장애물 감지 신호가 발생하며, 이를 차량 입차 요청 이벤트로 처리한다.

금속기어 서보모터 1개는 차단기 역할을 수행한다. STM32의 PWM 출력 신호를 이용하여 차단기의 열림/닫힘 각도를 제어한다. 서보모터 전원은 STM32 보드 전원과 분리하여 AA 4구 배터리 홀더 또는 별도 5V 전원을 사용한다. 이때 서보모터 전원 GND와 STM32 GND는 공통으로 연결한다.

1602 I2C Character LCD는 주차장 상태 표시용으로 사용한다. I2C 방식으로 연결하여 GPIO 사용량을 줄이고, 16자 x 2줄의 제한된 표시 공간을 고려하여 첫 번째 줄에는 빈자리 수와 주차칸 상태를 표시하고, 두 번째 줄에는 차단기 상태와 만차 여부를 표시한다. LCD 표시 기능은 별도의 `DisplayTask`에서 담당하도록 구성한다. I2C LCD backpack 주소는 모듈에 따라 `0x27` 또는 `0x3F`일 수 있으므로, 구현 단계에서 실제 주소를 확인한다.

2색 LED 모듈은 입차 가능 여부를 직관적으로 표시하는 데 사용한다. 초록색은 입차 가능, 빨간색은 만차 또는 입차 불가 상태로 정의한다.

수동 부저 모듈은 만차 상태에서 차량이 접근했을 때 경고음을 출력하거나 차단기 개폐 시 짧은 동작 피드백을 제공하는 데 사용한다. PWM 또는 GPIO 제어를 통해 경고음 패턴을 구현한다.

버튼 모듈은 출차 이벤트 및 관리자 수동 제어 기능에 사용한다. 버튼 입력은 차량이 출차하려는 상황을 알리는 이벤트로만 사용하고, 주차칸 점유 상태는 초음파 센서 측정 결과로만 결정한다. 필요 시 시스템 상태를 초기화하는 기능도 구현할 수 있다.

## 4. 소프트웨어 설계

### 4.1 RTOS 사용 이유

본 프로젝트는 주차칸 감지, 입구 차량 감지, 버튼 기반 출차 이벤트 처리, 차단기 제어, LCD 표시, 경고음 출력, USART 로그 출력 등 여러 기능이 동시에 동작해야 한다. 이를 하나의 `while` 루프로 구현할 경우, 특정 기능의 지연이 다른 기능의 응답성을 떨어뜨릴 수 있다. 예를 들어 초음파 센서 측정 중 대기 시간이 길어지면 입구 차량 감지나 차단기 제어가 늦어질 수 있고, LCD 갱신 중에는 경고음 출력이나 상태 판단이 지연될 수 있다.

uC/OS-III를 사용하면 기능별로 태스크를 분리할 수 있으며, 각 태스크에 서로 다른 우선순위를 부여할 수 있다. 입구 차량 감지와 차단기 제어처럼 빠른 반응이 필요한 기능은 높은 우선순위로 처리하고, LCD 표시나 USART 로그 출력처럼 상대적으로 지연되어도 되는 기능은 낮은 우선순위로 처리한다. 또한 메시지 큐와 세마포어를 이용하여 태스크 간 데이터를 명확하게 전달하고, 임계 구역을 이용하여 공유 데이터의 동시 접근 문제를 방지할 수 있다.

따라서 본 프로젝트는 RTOS 기반 태스크 분리, 우선순위 설계, 태스크 간 통신, 상태 머신 설계를 실습하기에 적절한 주제이다.

### 4.2 태스크 구성

본 프로젝트는 7개의 태스크로 구성한다. `ButtonTask`는 출차 이벤트를 버튼으로 입력받기 위해 기본 구성에 포함한다.

| Task | 역할 | 입력 | 출력 또는 전달 대상 |
|---|---|---|---|
| `SlotSensorTask` | 2개의 HC-SR04 센서를 순차 구동하여 주차칸 거리 측정 | 초음파 센서 Echo | `SensorQueue` |
| `EntranceTask` | 적외선 센서로 입구 차량 접근 감지 | 적외선 장애물 감지 센서 | `EntranceSem` 또는 Event Flag |
| `ParkingManagerTask` | 주차칸 상태 판단, 빈자리 수 계산, 전체 상태 관리 | `SensorQueue`, `EntranceSem` | `GateCommandQueue`, `DisplayQueue`, `LogQueue` |
| `GateControlTask` | 서보모터로 차단기 열림/닫힘 제어 | `GateCommandQueue` | 서보모터 PWM |
| `DisplayTask` | LCD와 2색 LED 갱신 | `DisplayQueue`, 공유 상태 | 1602 I2C LCD, 2색 LED |
| `AlarmLogTask` | 부저 출력과 USART 로그 출력 | `LogQueue`, 알람 이벤트 | 부저, USART |
| `ButtonTask` | 버튼 디바운싱 및 출차 이벤트/관리자 수동 제어 | 버튼 모듈 | 출차 이벤트, 관리자 이벤트 |

`SlotSensorTask`는 2개의 HC-SR04 초음파 센서를 순차적으로 구동하여 각 주차칸의 거리값을 측정한다. 초음파 센서는 동시에 Trigger할 경우 센서 간 간섭이 발생할 수 있으므로, 각 센서를 시간 분할 방식으로 순차 측정한다. 측정된 거리값은 `ParkingManagerTask`로 메시지 큐를 통해 전달한다. 향후 주차칸을 4개로 확장할 경우 같은 task에서 센서 배열의 크기만 늘려 순차 측정 대상을 추가한다.

`EntranceTask`는 적외선 장애물 감지 센서를 통해 입구 차량 접근 여부를 확인한다. 차량 접근이 감지되면 세마포어 또는 이벤트 플래그를 이용하여 `ParkingManagerTask`에 입차 요청 이벤트를 전달한다.

`ParkingManagerTask`는 시스템의 핵심 상태 관리 태스크이다. `SlotSensorTask`로부터 받은 각 주차칸 거리값을 이용하여 `EMPTY` 또는 `OCCUPIED` 상태를 결정하고, 전체 빈자리 수를 계산한다. 또한 입구 차량 접근 이벤트가 발생했을 때 빈자리 여부를 판단하여 `GateControlTask`에 차단기 개방 또는 입차 거부 이벤트를 전달한다. 버튼 출차 이벤트가 발생하면 출차 요청 로그와 필요 동작을 처리하지만, 주차칸 상태는 직접 변경하지 않는다. 실제 빈자리 수는 이후 초음파 센서 측정 결과가 `EMPTY`로 바뀔 때 갱신된다.

`GateControlTask`는 서보모터를 이용하여 차단기 열림/닫힘 상태를 제어한다. 입차 허용 이벤트를 받으면 차단기를 열고, 일정 시간이 지나면 차단기를 닫는다. 만차 상태에서는 차단기를 닫은 상태로 유지한다.

`DisplayTask`는 1602 I2C Character LCD와 2색 LED를 제어한다. 현재 빈자리 수, 각 주차칸 상태, 차단기 상태를 LCD에 요약 표시하고, 입차 가능/만차 여부를 2색 LED로 표시한다. LCD는 16자 x 2줄로 표시 공간이 작으므로 자세한 센서값과 이벤트 기록은 USART 로그로 출력한다. I2C LCD는 공유 자원이므로 LCD 접근 시에는 뮤텍스 또는 임계 구역을 사용한다.

`AlarmLogTask`는 수동 부저와 USART 로그 출력을 담당한다. 만차 상태에서 차량 접근이 감지되면 부저로 경고음을 출력하고, 주요 상태 변화와 이벤트를 USART 로그로 출력한다.

`ButtonTask`는 버튼 디바운싱을 수행하고, 출차 이벤트 또는 관리자 이벤트를 `ParkingManagerTask`에 전달한다. 실제 출구 센서를 추가하지 않는 대신 버튼을 통해 출차 상황을 알리되, 주차칸 점유 여부는 초음파 센서가 계속 담당한다.

## 5. 태스크 우선순위 설계

각 태스크의 우선순위는 실시간 응답 필요성에 따라 다르게 설정한다.

가장 높은 우선순위는 `EntranceTask` 또는 `ButtonTask`에 부여한다. 차량 입차 요청이나 출차 버튼 이벤트는 시스템 반응이 빠르게 이루어져야 하므로 높은 우선순위가 필요하다.

그 다음 우선순위는 `GateControlTask`에 부여한다. 차단기 개폐는 사용자에게 직접 보이는 액추에이터 동작이며, 입차 허용 또는 거부 결정 이후 지연이 적어야 한다.

`ParkingManagerTask`는 센서값과 이벤트를 기반으로 전체 상태를 판단하므로 중간 이상의 우선순위를 갖는다.

`SlotSensorTask`는 주기적으로 실행되는 태스크이며, 여러 초음파 센서를 순차적으로 측정한다. 일정한 주기로 실행되면 되므로 중간 우선순위로 설정한다.

`DisplayTask`와 `AlarmLogTask`는 상대적으로 지연 허용 범위가 크므로 낮은 우선순위로 설정한다. LCD 표시와 USART 로그는 시스템 동작 확인에 중요하지만, 차단기 제어나 차량 접근 감지보다 우선되어서는 안 된다.

예상 우선순위는 다음과 같다.

| Task | 우선순위 예시 | 우선순위 근거 |
|---|---:|---|
| `EntranceTask` | 5 | 입구 차량 감지의 빠른 반응 필요 |
| `ButtonTask` | 6 | 출차 이벤트 및 관리자 수동 조작 반응 필요 |
| `GateControlTask` | 7 | 차단기 액추에이터 제어 지연 최소화 |
| `ParkingManagerTask` | 8 | 센서값과 이벤트 기반 상태 판단 |
| `SlotSensorTask` | 9 | 주기적 센서 측정 |
| `DisplayTask` | 10 | 표시 갱신은 지연 허용 가능 |
| `AlarmLogTask` | 11 | 로그 출력은 가장 낮은 우선순위로 처리 가능 |

실제 구현에서는 수업 코드의 우선순위 설정 방식과 기존 태스크 우선순위와의 충돌 여부를 고려하여 조정한다.

## 6. 태스크 간 통신 구조

본 시스템에서는 전역 변수만을 사용하는 방식 대신 uC/OS-III의 태스크 간 통신 기능을 사용한다.

| 송신 Task | 수신 Task | 통신 객체 | 전달 데이터 |
|---|---|---|---|
| `SlotSensorTask` | `ParkingManagerTask` | `SensorQueue` | 센서 번호, 거리값 |
| `EntranceTask` | `ParkingManagerTask` | `EntranceSem` 또는 Event Flag | 입구 차량 접근 이벤트 |
| `ParkingManagerTask` | `GateControlTask` | `GateCommandQueue` | `GATE_OPEN`, `GATE_CLOSE`, `GATE_DENY` |
| `ParkingManagerTask` | `DisplayTask` | `DisplayQueue` | 빈자리 수, 주차칸 상태, 차단기 상태 |
| `ParkingManagerTask` | `AlarmLogTask` | `LogQueue` | 이벤트 코드 또는 로그 메시지 |
| `ButtonTask` | `ParkingManagerTask` | Event Flag 또는 Queue | 출차 이벤트, 초기화 이벤트 |

`SlotSensorTask`는 각 주차칸의 센서 번호와 거리값을 구조체 형태로 만들어 `SensorQueue`에 post한다. `ParkingManagerTask`는 `SensorQueue`에서 데이터를 pend하여 주차칸 상태를 갱신한다.

`EntranceTask`는 차량 접근이 감지되면 `EntranceSem` 또는 Event Flag를 post한다. `ParkingManagerTask`는 해당 이벤트를 수신한 뒤 현재 빈자리 수를 확인하고, 입차 허용 여부를 판단한다.

`ParkingManagerTask`는 `GateControlTask`로 `GateCommandQueue`를 통해 명령을 전달한다. 명령은 `GATE_OPEN`, `GATE_CLOSE`, `GATE_DENY`와 같은 형태로 정의한다.

`ParkingManagerTask`는 `DisplayTask`로 `DisplayQueue`를 통해 현재 시스템 상태를 전달한다. `DisplayTask`는 이를 바탕으로 LCD와 LED를 갱신한다.

`AlarmLogTask`는 `LogQueue`를 통해 주요 이벤트 문자열 또는 로그 코드를 전달받아 USART로 출력한다. 만차 상태에서 차량 접근이 발생한 경우에는 부저 출력도 함께 수행한다.

`ButtonTask`는 출차 버튼이 눌렸을 때 출차 이벤트를 `ParkingManagerTask`에 전달한다. `ParkingManagerTask`는 출차 요청을 로그로 남기고 필요한 출력 동작을 수행하지만, 특정 주차칸을 직접 비우지는 않는다. 빈자리 수와 표시 정보는 다음 초음파 센서 측정 결과를 기준으로 갱신한다.

공유 데이터인 parking state, empty count, gate state 등은 임계 구역을 이용해 보호한다. 특히 여러 태스크가 같은 상태 구조체를 읽거나 갱신할 수 있으므로, `CPU_CRITICAL_ENTER()`와 `CPU_CRITICAL_EXIT()`를 사용하여 동시 접근 문제를 방지한다.

## 7. 상태 머신 설계

본 시스템은 `ParkingManagerTask`를 중심으로 상태 머신을 구성한다. 주요 상태는 다음과 같다.

| 상태 | 의미 | 주요 동작 |
|---|---|---|
| `INIT` | 시스템 초기화 단계 | 센서, LCD, 서보모터, USART, RTOS 객체 초기화 |
| `NORMAL` | 기본 동작 상태 | 주차칸 상태 갱신, LCD 표시 |
| `CAR_DETECTED` | 입구 차량 접근 감지 | 현재 빈자리 수 확인 |
| `GATE_OPEN` | 차단기 개방 상태 | 일정 시간 후 차단기 닫힘 처리 |
| `FULL` | 빈자리 없음 | 차단기 닫힘 유지, LED와 부저로 입차 불가 알림 |
| `ERROR` | 센서 오류 또는 비정상 측정 | LCD와 USART 로그에 오류 표시, 재측정 시도 |

주요 상태 전환은 다음과 같다.

```text
INIT -> NORMAL: 초기화 완료
NORMAL -> CAR_DETECTED: 입구 차량 감지
CAR_DETECTED -> GATE_OPEN: 빈자리 있음
CAR_DETECTED -> FULL: 빈자리 없음
GATE_OPEN -> NORMAL: 차단기 닫힘 완료
NORMAL -> NORMAL: 출차 버튼 이벤트 기록 후 초음파 센서 결과로 빈자리 수 갱신
NORMAL -> ERROR: 센서 오류 또는 비정상 측정
ERROR -> NORMAL: 오류 복구 완료
```

## 8. USART 로그 설계

USART 로그는 동작 동영상과 최종 보고서에서 태스크 동작을 확인하는 핵심 수단으로 사용한다. 주요 로그 형식은 다음과 같다.

```text
[SENSOR] slot=1 distance=12.4cm state=OCCUPIED
[SENSOR] slot=2 distance=38.1cm state=EMPTY
[STATE] empty=1/2 gate=CLOSED
[EVENT] entrance_detected
[EVENT] exit_button_pressed
[GATE] open
[GATE] close
[ALARM] parking_full entrance_denied
[ERROR] sensor_timeout slot=2
[BUTTON] exit_event_logged
```

USART 로그를 통해 센서 측정, 상태 판단, 이벤트 전달, 차단기 제어가 실제로 독립적인 태스크에서 수행되고 있음을 확인할 수 있도록 한다.

## 9. RTOS를 사용하지 않았을 때의 문제점과 uC/OS-III 적용 효과

본 시스템을 단일 `while` 루프로 구현할 경우, 초음파 센서 측정, 입구 감지, 버튼 출차 이벤트 처리, LCD 표시, 부저 출력, 서보모터 제어, USART 로그 출력이 모두 순차적으로 실행된다. 이 경우 특정 작업이 지연되면 전체 시스템 반응이 늦어질 수 있다. 예를 들어 LCD 갱신이나 USART 출력이 길어지면 입구 차량 감지 처리가 늦어지고, 초음파 센서 측정 대기 시간이 길어지면 차단기 제어가 지연될 수 있다. 또한 모든 상태를 전역 변수로 관리하면 코드가 복잡해지고, 동시 접근에 따른 데이터 불일치 문제가 발생할 수 있다.

uC/OS-III를 사용하면 각 기능을 독립적인 태스크로 분리할 수 있다. 입구 감지와 차단기 제어처럼 빠르게 반응해야 하는 기능은 높은 우선순위로 처리하고, LCD 표시와 로그 출력처럼 지연이 허용되는 기능은 낮은 우선순위로 처리한다. 또한 메시지 큐와 세마포어를 통해 태스크 간 데이터를 명확하게 전달할 수 있고, 임계 구역을 통해 공유 데이터 접근을 보호할 수 있다.

따라서 본 프로젝트는 RTOS의 장점인 태스크 분리, 우선순위 기반 스케줄링, 이벤트 기반 처리, 태스크 간 통신, 공유 자원 보호를 실습할 수 있는 구조를 가진다.

## 10. 예상 동작 시나리오

시스템이 시작되면 LCD에 `SMART PARKING`을 짧게 표시한 뒤, 현재 빈자리 수와 주차칸 상태를 16자 x 2줄 형식으로 표시한다. 예를 들어 첫 번째 줄에는 `FREE:1/2 S1:O`, 두 번째 줄에는 `S2:E G:CLOSED`처럼 요약 정보를 표시한다. `SlotSensorTask`는 2개의 초음파 센서를 순차적으로 측정하여 각 주차칸의 점유 상태를 갱신한다. `ParkingManagerTask`는 이를 기반으로 빈자리 수를 계산하고 `DisplayTask`는 LCD에 상태를 표시한다. 코드 구조는 주차칸 수를 상수로 관리하므로, 센서와 핀 정의를 추가하면 4개 주차칸까지 확장할 수 있다.

차량 모형이 입구에 접근하면 적외선 장애물 감지 센서가 이를 감지한다. `EntranceTask`는 차량 접근 이벤트를 발생시키고, `ParkingManagerTask`는 현재 빈자리 수를 확인한다. 빈자리가 있으면 `GateControlTask`에 차단기 개방 명령을 전달한다. `GateControlTask`는 서보모터를 동작시켜 차단기를 열고, 일정 시간이 지난 뒤 차단기를 닫는다. 이 과정에서 USART 로그에는 입구 감지, 입차 허용, 차단기 열림/닫힘 이벤트가 출력된다.

반대로 모든 주차칸이 점유된 상태에서 차량이 입구에 접근하면 `ParkingManagerTask`는 만차 상태로 판단한다. 이 경우 `GateControlTask`에는 차단기 닫힘 유지 명령이 전달되고, `DisplayTask`는 LCD와 2색 LED를 통해 만차 상태를 표시한다. `AlarmLogTask`는 수동 부저를 울리고 USART 로그에 입차 거부 이벤트를 출력한다.

차량이 출차하려는 상황에서는 관리자 버튼을 누른다. `ButtonTask`가 버튼 입력을 감지하고 `ParkingManagerTask`에 출차 이벤트를 전달하면, `ParkingManagerTask`는 출차 이벤트를 로그로 남기고 필요한 출력 동작을 수행한다. 이후 차량 모형이 실제로 주차칸에서 제거되면 `SlotSensorTask`의 다음 거리 측정 결과가 달라지고, `ParkingManagerTask`는 해당 주차칸을 `EMPTY`로 판단하여 빈자리 수를 갱신한다. 이렇게 하면 점유 상태의 기준을 초음파 센서로 일원화할 수 있다.

## 12. 기대 효과 및 결론

본 프로젝트를 통해 uC/OS-III 기반 임베디드 시스템에서 여러 기능을 태스크로 분리하고, 우선순위와 태스크 간 통신을 설계하는 경험을 얻을 수 있다. 주차장 빈자리 감지, 차량 입차 제어, 버튼 기반 출차 이벤트 처리라는 직관적인 시나리오를 통해 센서 입력, 상태 판단, 액추에이터 제어, 사용자 표시, 알람, 로그 출력이 동시에 수행되는 구조를 구현할 수 있다.

특히 본 프로젝트는 단순한 센서 확인 예제가 아니라, 주기적으로 실행되는 센서 측정 태스크와 이벤트 발생 시 동작하는 입구 감지 및 차단기 제어 태스크를 분리한다는 점에서 RTOS 활용 목적이 분명하다. 또한 메시지 큐, 세마포어, 임계 구역, 상태 머신, USART 로그를 포함하여 실시간 임베디드 소프트웨어 설계의 핵심 요소를 실습할 수 있다.

최종적으로 본 시스템은 소형 모형 주차장을 대상으로 2개 주차칸의 차량 점유 상태를 감지하고, 빈자리가 있을 때만 차량 입차를 허용하는 동작을 시연한다. 출차 버튼은 출차 이벤트 입력과 로그 용도로 사용하며, 실제 점유 상태는 초음파 센서 측정값으로만 관리한다. 설계 구조는 4개 주차칸까지 확장 가능하도록 유지한다. 이를 통해 RTOS 기반 임베디드 시스템의 구조적 장점과 실시간 제어 방식의 필요성을 확인할 수 있을 것으로 기대한다.

## 13. 기획서 분석 및 보완 제안

### 13.1 과제 요구사항 충족 여부

### 13.2 강점

- 주제가 직관적이라 동작 영상에서 결과를 보여주기 쉽다.
- 입력, 판단, 출력, 표시, 로그가 자연스럽게 분리되어 RTOS task 설계의 명분이 충분하다.
- 단순 센서 읽기 예제가 아니라 queue, semaphore, 상태 머신을 실제 동작 흐름에 사용할 수 있다.
- 만차 상태, 입차 허용, 버튼 출차 이벤트, 센서 기반 빈자리 갱신, 센서 오류 등 정상/예외 시나리오를 만들기 쉽다.
- USART 로그를 설계에 포함한 점은 보고서와 동작 검증에 유리하다.

### 13.3 위험 요소

- 기본 구현은 HC-SR04 2개로 줄였지만, LCD, 서보모터, 부저, LED, 버튼, 적외선 센서를 모두 붙이면 여전히 배선과 핀 배치 확인에 시간이 들어갈 수 있다.
- HC-SR04를 안정적으로 측정하려면 echo pulse width 측정 방식이 필요하다. 단순 blocking delay로 구현하면 RTOS task 분리의 장점이 약해질 수 있다.
- 서보모터 전원은 반드시 별도 전원과 공통 GND를 사용해야 한다. 보드 5V만으로 구동하면 리셋이나 오동작이 생길 수 있다.
- 1602 I2C LCD는 표시 공간이 16자 x 2줄로 제한되고, 모듈별 I2C 주소 확인이 필요하다. 최종 평가에서는 LCD보다 task 구조와 동작성이 더 중요하므로, LCD 문구는 요약형으로 유지하고 자세한 정보는 USART 로그와 LED 표시로 보완해야 한다.
- task 수가 많아질수록 디버깅 난이도가 올라간다. 구현이 늦어지면 LCD 표시나 로그 출력 기능을 단순화하고, 입차 감지, 출차 버튼 이벤트, 초음파 기반 상태 판단, 차단기 제어를 먼저 완성하는 편이 안전하다.

### 13.4 보완 제안

- 구현 1차 목표는 주차칸 2개로 end-to-end 동작을 완성하는 것이다. 이후 센서 배열과 상태 배열을 확장하여 4개 주차칸까지 늘릴 수 있음을 보고서에 명시한다. 단, 1602 I2C LCD는 표시 공간이 제한되므로 4칸 확장 시에는 LCD에 요약 정보만 표시하고 상세 상태는 USART 로그로 확인한다.
- `SlotSensorTask`는 모든 센서를 한 번에 처리하되, 센서별 측정 결과만 queue로 보내는 구조가 적절하다. 센서마다 task를 따로 만들 필요는 없다.
- `ParkingManagerTask`가 상태의 단일 소유자가 되도록 설계하는 것이 좋다. 다른 task는 상태를 직접 수정하지 않고 queue나 semaphore로 요청만 전달하게 하면 race condition이 줄어든다.
- 보고서에는 전체 코드보다 task 생성 코드, queue post/pend 코드, 상태 머신 코드, critical section 사용 부분만 발췌하는 것이 좋다.
- 동작 영상은 `정상 입차 -> 만차 거부 -> 버튼 출차 이벤트 -> 차량 제거 후 초음파 기반 빈자리 갱신 -> USART 로그 확인` 순서로 촬영하면 평가자가 RTOS 구조를 이해하기 쉽다.
